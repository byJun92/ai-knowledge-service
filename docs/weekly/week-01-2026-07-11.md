## 4. 핵심 개념

### 4.1 `uv`

`uv`는 Python 버전, 가상환경, 의존성, 잠금 파일과 명령 실행을 한 프로젝트에서 관리하기 위한 도구다.

이번 프로젝트에서 선택한 이유:

- `pyproject.toml`과 `uv.lock`으로 환경을 재현하기 쉽다.
- macOS에서 설치가 간단하다.
- 이후 Docker와 GitHub Actions에서도 같은 명령을 사용하기 쉽다.
- `pip`, `venv`, 일부 `pyenv` 역할을 한 흐름으로 관리할 수 있다.

공식 문서:
- https://docs.astral.sh/uv/getting-started/installation/

### 4.2 `src` layout

애플리케이션 코드를 저장소 루트가 아니라 `src/<package>` 아래에 둔다.

이 구조는 테스트 중 현재 디렉터리의 파일이 우연히 import되는 문제를 줄이고, 실제 패키지 설치 상태와 비슷한 조건에서 검증하게 한다.

### 4.3 타입 힌트

Python 타입 힌트는 Java의 컴파일 타입 검사와 동일하지 않다. 실행 시 강제되지 않는 경우가 많지만 다음 효과가 있다.

- IDE 자동완성 향상
- 함수 입출력 계약 표현
- 리팩터링 안정성 향상
- 이후 Pydantic과 FastAPI 스키마의 기반

### 4.4 pytest

pytest는 단순한 `assert` 기반 테스트와 자동 테스트 탐색을 제공한다. 공식 문서 기준으로 `test_*.py` 또는 `*_test.py` 형태의 파일을 자동 탐색할 수 있다.

공식 문서:
- https://docs.pytest.org/en/stable/getting-started.html

### 4.5 Ruff

Ruff는 Python linter와 formatter를 한 도구로 제공한다. 이번 프로젝트에서는 초기 도구 수를 줄이고 일관된 검사 명령을 유지하기 위해 사용한다.

공식 문서:
- https://docs.astral.sh/ruff/

## Step 6. 기본 도메인 모델 작성

`src/ai_knowledge_service/domain/document.py`

```python
from dataclasses import dataclass
from datetime import UTC, datetime
from enum import StrEnum


class DocumentType(StrEnum):
    PAPER = "paper"
    TECH_ARTICLE = "tech_article"
    DOCUMENTATION = "documentation"


@dataclass(frozen=True, slots=True)
class Document:
    document_id: str
    title: str
    source_url: str
    document_type: DocumentType
    collected_at: datetime

    @classmethod
    def create(
        cls,
        *,
        document_id: str,
        title: str,
        source_url: str,
        document_type: DocumentType,
    ) -> "Document":
        normalized_id = document_id.strip()
        normalized_title = title.strip()
        normalized_url = source_url.strip()

        if not normalized_id:
            raise ValueError("document_id must not be blank")
        if not normalized_title:
            raise ValueError("title must not be blank")
        if not normalized_url.startswith(("https://", "http://")):
            raise ValueError("source_url must be an HTTP(S) URL")

        return cls(
            document_id=normalized_id,
            title=normalized_title,
            source_url=normalized_url,
            document_type=document_type,
            collected_at=datetime.now(UTC),
        )
```

### 설계 판단

- `frozen=True`: 생성 후 값 변경을 막는다. Java의 불변 DTO와 유사하다.
- `slots=True`: 허용되지 않은 속성이 동적으로 생기는 것을 막고 메모리 사용도 줄인다.
- `StrEnum`: 문자열 저장·직렬화가 쉬우면서 허용 값은 제한한다.
- `create`: 검증과 정규화를 생성 경로 한 곳에 모은다.

## Step 7. 테스트 작성

`tests/domain/test_document.py`

```python
import pytest

from ai_knowledge_service.domain.document import Document, DocumentType


def test_create_document() -> None:
    document = Document.create(
        document_id="paper-001",
        title="Retrieval-Augmented Generation",
        source_url="https://example.com/paper",
        document_type=DocumentType.PAPER,
    )

    assert document.document_id == "paper-001"
    assert document.title == "Retrieval-Augmented Generation"
    assert document.document_type is DocumentType.PAPER
    assert document.collected_at.tzinfo is not None


@pytest.mark.parametrize(
    ("document_id", "title", "source_url", "expected_message"),
    [
        ("", "Valid title", "https://example.com", "document_id"),
        ("doc-1", " ", "https://example.com", "title"),
        ("doc-1", "Valid title", "ftp://example.com", "source_url"),
    ],
)
def test_create_document_rejects_invalid_input(
    document_id: str,
    title: str,
    source_url: str,
    expected_message: str,
) -> None:
    with pytest.raises(ValueError, match=expected_message):
        Document.create(
            document_id=document_id,
            title=title,
            source_url=source_url,
            document_type=DocumentType.DOCUMENTATION,
        )
```

## 10. 최신 AI 논문 브리핑

## Large Language Models for Software Engineering: A Reproducibility Crisis

- 저자: Mohammed Latif Siddiq, Arvin Islam-Gomes, Natalie Sekerak, Joanna C. S. Santos
- arXiv 등록일: 2025-11-29
- 원문: https://arxiv.org/abs/2512.00651

### 선정 이유

이번 주는 모델 성능보다 **재현 가능한 개발 환경과 저장소 품질**이 핵심이다. 이 논문은 LLM 기반 소프트웨어 엔지니어링 연구의 코드, 데이터, 문서, 실행 환경, 버전 관리 문제를 대규모로 분석하므로 Week 01과 직접 연결된다.

### 연구 배경

LLM 기반 소프트웨어 엔지니어링 연구가 빠르게 증가했지만, 공개된 코드와 논문만으로 동일 결과를 재현할 수 있는지는 충분히 검증되지 않았다.

### 핵심 아이디어와 방법

연구진은 2017년부터 2025년까지 발표된 LLM-for-SE 논문 640편과 연관 저장소·문서를 분석했다.

다음 7개 범주의 재현성 문제를 분류했다.

1. Code and Execution
2. Data
3. Documentation
4. Environment and Tooling
5. Versioning
6. Model
7. Access and Legal

### 주요 결과

논문이 보고한 핵심 결과는 다음과 같다.

- 실행 환경 명세가 부족한 사례가 지속적으로 존재했다.
- 버전 관리와 문서화 품질에도 반복적인 문제가 있었다.
- artifact badge는 산출물 존재를 나타낼 수 있지만, 실제 실행 충실도나 장기 재현성을 항상 보장하지는 않았다.
- 연구진은 이진적인 “산출물 있음/없음”보다 다차원적인 Reproducibility Maturity Model을 제안했다.

### 한계

- 논문과 공개 저장소를 중심으로 판단하므로 비공개 내부 환경은 반영하기 어렵다.
- 재현성은 모델 API 종료, 라이선스, 데이터 접근 제한 같은 외부 요인에도 영향을 받는다.
- 연구 결과가 일반 AI 애플리케이션 전체에 동일하게 적용된다고 단정할 수는 없다.

### 프로젝트 적용점

이번 프로젝트에서는 처음부터 다음을 지킨다.

- `uv.lock` 커밋
- Python 버전 고정
- 실행 명령 README 기록
- 테스트 자동화
- 의존성·도구 설정을 `pyproject.toml`에 집중
- API Key를 저장소에서 제외
- 주차별 문서와 ADR 기록
- 이후 모델명과 프롬프트 버전 기록
- 평가 데이터셋과 평가 결과 버전 관리

### 사실과 해석 구분

- 사실: 논문은 640편을 분석하고 재현성 문제 범주와 성숙도 모델을 제시한다.
- 해석: 이 포트폴리오에서 재현성 자료를 초기에 구축하면 단순 데모보다 실제 엔지니어링 역량을 보여주는 데 유리하다.
